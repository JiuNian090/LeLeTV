LeLeTV 项目架构重构讨论总结

日期：2026-06-12
参与角色：资深架构师（AI）、项目负责人
目标：在不影响用户体验和视觉效果的前提下，对现有 LeLeTV 项目进行架构现代化重构，并为未来引入 Vue 等框架做好准备。

***

一、现状分析

1.1 项目整体评价

· LeLeTV 是一个个人影视聚合项目，核心代码约 800 行。
· 架构设计清晰，亮点突出（如负载均衡、缓存管理、模块拆分）。
· 已达到优秀个人项目标准，但从工程化角度看仍有提升空间。

1.2 当前技术栈与结构

· 纯原生 JavaScript（ES5/ES6 风格），无构建工具。
· 文件按功能划分：js/api/、js/core/、js/player/、js/ui/ 等。
· 页面模型：两个独立页面（主页面 index.html + 播放页面 player.html）。
· 主页面内导航栏切换面板：采用原生 JS + CSS 显示/隐藏（多面板切换，不依赖 URL 路由）。
· 播放器基于 ArtPlayer + HLS.js，手动封装。

1.3 存在的隐患（需提前优化）

· 大量全局变量/函数挂在 window 下。
· 事件监听未统一解绑，可能造成内存泄漏或重复绑定。
· DOM 选择器硬编码层级，重构易失效。
· 模块初始化顺序依赖 <script> 标签顺序。
· localStorage 键名散落各处，无统一管理。
· 播放器实例全局引用，销毁/重建逻辑不严谨。
· CSS 布局对绝对定位/固定高度有隐式依赖。

***

二、重构目标与原则

2.1 核心目标

· 用户体验零变化：UI、交互、视觉效果完全一致。
· 性能更优：按需加载、热更新、Tree Shaking。
· 代码更清晰：模块化、类型安全、统一状态管理。
· 易于维护：依赖清晰、可测试、可扩展。

2.2 关键原则

· 分阶段实施：先换引擎（架构迁移），再换内饰（Vue 组件库）。
· 绞杀者模式：新系统逐步替代旧系统，用户无感知。
· 保留所有现有 UI 结构：DOM id/class、CSS 样式、事件绑定方式不变。

***

三、推荐的技术演进路线

阶段一：架构迁移（仅底层技术栈升级，UI/功能不变）

目标：使用现代工程化工具重构代码结构，但不改变任何界面行为。

技术选型：

· 构建工具：Vite（极速热更新，按需编译）
· 语言：TypeScript（渐进式增强，兼容现有 JS）
· 模块系统：ES Modules + 简单的依赖注入容器（或 provide/inject）
· 状态管理：临时使用极简单例对象（第二阶段替换为 Pinia）
· 事件通信：极简事件总线（第二阶段替换为 mitt）

文件结构（阶段一输出）：

```
src/
├── main.ts                 # 入口，初始化顺序显式控制
├── core/
│   ├── di.ts              # 轻量级依赖注入
│   ├── eventBus.ts        # 全局事件总线
│   └── store.ts           # 简单状态单例
├── services/              # 业务逻辑层（无 DOM 操作）
│   ├── api/               # 原 api.js 拆分
│   ├── player/            # 原 player-core.js 拆分
│   ├── auth.ts
│   ├── cache.ts
│   └── history.ts
├── ui/                    # UI 操作层（保留原 DOM 操作）
│   ├── components/        # 可复用 UI 模块（仍用原生 JS）
│   └── layouts/
├── types/                 # TypeScript 类型定义
├── assets/                # 原 css/, fonts/, images/
└── index.html             # 入口（仅修改 script 标签）
```

页面切换机制：

· 主页面内面板切换：保持 hash 路由改造（见后文优化）。
· 主页面 ↔ 播放页面：维持传统的 window\.location.href 跳转，完全不受影响。

阶段二：渐进式引入 Vue（UI 逐步组件化）

目标：在不影响整体功能的前提下，逐步将原生 UI 替换为 Vue 组件，最终完全由 Vue 接管。

技术选型：

· 框架：Vue 3（组合式 API）
· 状态管理：Pinia（替代阶段一的 store）
· 路由：Vue Router（hash 模式，与阶段一 hash 路由无缝衔接）
· UI 组件库（可选）：Element Plus、Vuetify 等，按需引入，替换原生组件。

迁移策略：

· 使用双架构并存（Vue 实例与原生模块共存）。
· 从最独立、最简单的组件开始替换（如“关于面板”）。
· 播放器等复杂模块先封装成无框架类，再包装为 Vue 组件。
· 最后移除所有原生 UI 代码，根节点交由 Vue 管理。

四、在重构前需要立即优化的“雷区”

为避免重构过程出现功能丢失或大改，建议先执行以下优化（对现有功能零影响）：

优化项 具体做法 优先级
全局变量收敛 创建 window\.LeLeTV 适配器，将所有跨模块访问的变量/函数挂载到此对象下 ⭐⭐⭐⭐⭐
存储服务统一 封装 storageService.js，统一管理所有 localStorage 键名和读写方法 ⭐⭐⭐⭐
播放器管理器 实现 playerManager.js，提供 create() / destroy() / getInstance()，确保单例及正确销毁 ⭐⭐⭐⭐⭐
事件解绑机制 为每个动态模块提供 destroy() 函数，在模块移除时解绑事件 ⭐⭐⭐⭐
选择器硬化 使用 data-\* 属性代替层级选择器，例如 \[data-video-id] ⭐⭐⭐
时间常量集中 将 setTimeout、防抖节流等时间值提取到 config/timing.js ⭐⭐
初始化顺序显式化 将所有初始化代码放在 main.ts 中按依赖顺序调用，不依赖脚本标签顺序 ⭐⭐⭐⭐
CSS 布局弹性化 用 flex/grid 替代绝对定位，使用 CSS 变量定义尺寸 ⭐⭐

五、导航栏切换模式的改进建议

5.1 现有模式的问题

· 主页面内“搜索/历史/设置/关于”等面板切换不改变 URL。
· 导致无法直接链接到某个面板、浏览器前进/后退无效、刷新后状态丢失。

5.2 推荐改进：Hash 路由模式

优点：

· URL 变化（#search、#history），浏览器历史记录正常。
· 前进/后退按钮有效。
· 刷新页面后仍停留在当前面板。
· 未来迁移至 Vue Router 时只需替换底层实现，上层逻辑几乎不改。

实现要点（保留原视觉效果）：

```javascript
// 监听 hashchange 事件，根据 hash 显示对应面板
function handleHashChange() {
  let panel = location.hash.slice(1) || 'search';
  switchToPanel(panel); // 与原显示/隐藏逻辑一致
}
window.addEventListener('hashchange', handleHashChange);
```

该优化可在重构前独立完成，工作量极小（约 30 行代码），且不破坏现有功能。

***

六、迁移过程如何保证页面切换不出问题

· 主页面内面板切换：在阶段一完全保留原生显示/隐藏逻辑，仅将事件监听移至模块中，行为不变。
· 主页面 ↔ 播放页面：使用传统 URL 跳转，不受任何影响。
· 验证清单（迁移后逐一测试）：
· 点击导航菜单，面板正确切换。
· 浏览器前进/后退按钮能正确切换面板（hash 路由改造后）。
· 刷新页面，停留在当前面板（改造后）。
· 点击视频卡片，跳转到播放页且能正常播放。
· 播放页返回主页面，历史/设置等状态不丢失。

七、使用 AI Agent 辅助迁移的策略

7.1 为什么适合 AI

· 阶段一任务是机械性重构（全局转模块、添加类型、替换引用），规则明确，适合批量处理。
· 阶段二需要创造性转换（命令式 DOM → 声明式 Vue），AI 可先生成骨架，人工微调。

7.2 给 AI Agent 的指令模板

“对以下 JS 文件执行架构迁移，仅改动模块导入导出方式，不修改任何 DOM 操作、函数签名、全局变量引用。完成后输出新的 TS 文件，并保持原功能完全不变。”

八、结论与最终建议

1. 立即执行“重构前的 8 项优化”（尤其是全局适配器、存储服务、播放器管理器），为后续迁移扫清障碍。
2. 先对导航栏进行 hash 路由改造，提升当前用户体验，同时为 Vue Router 铺路。
3. 按阶段实施：阶段一（Vite + TS + 模块化）→ 阶段二（渐进引入 Vue）。
4. 充分利用 AI Agent 执行机械性重构，人工关注业务逻辑的正确性。
5. 整个过程中，用户无感知，视觉效果完全不变。
